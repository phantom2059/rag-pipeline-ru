# baseline_rag

# RAG Pipeline с LLM

Проект представляет собой полноценный RAG (Retrieval-Augmented Generation) pipeline для работы с русскоязычными вопросами и ответами. Система объединяет векторный поиск по базе знаний с генерацией ответов на основе языковой модели.

## 📋 Описание

Проект реализует гибридную систему ответов на вопросы, которая:
- Использует **RAG** для поиска релевантного контекста в базе знаний
- Генерирует ответы с помощью **квантованной LLM модели** (Mistral-7B-Saiga)
- Поддерживает **лемматизацию** для улучшения качества поиска на русском языке
- Имеет встроенную обработку **математических запросов**
- Работает как с GPU, так и с CPU (с автоматическим fallback)

## 🏗️ Архитектура

### Основные компоненты

1. **`factual_model.py`** — основной класс `FactualModel` с динамической загрузкой модели и RAG
2. **`model_downloader.py`** — функция `download_model()` (выбор модели, квантование 4/8 bit, авто-конфиг)
3. **`rag_builder.py`** — построение RAG-индекса из собственных данных (`create_rag_index()`)
4. **`utils.py`** — утилиты загрузки вопросов, сохранения ответов и генерации `config.json`
5. **`quick_start.ipynb`** — основной ноутбук с пошаговым пайплайном (установка, модель, RAG, инференс, вывод таблиц)
6. **`solution.py`** — CLI-обёртка для быстрого запуска без ноутбука
7. **`lem_worker.py`** — лемматизация русского языка
8. **`quantize_model.py`** — вспомогательные функции квантования (исторические)
9. **`saiga_benchmark.ipynb`** — экспериментальный ноутбук с метриками

### Структура данных RAG

RAG данные хранятся в папке `rag/` (или `artifacts/rag/`, `data/rag/` в зависимости от конфигурации) и включают:

- **`meta.json`** — метаданные (название embedding модели, параметры)
- **`index.faiss`** — векторный индекс FAISS для быстрого поиска
- **`chunks.parquet`** — текстовые чанки в формате Parquet (сжатые данные)

### Подготовка данных

1. **Скачивание модели**: вызов `download_model()` автоматически загружает веса (например, [`Gaivoronsky/Mistral-7B-Saiga`](https://huggingface.co/Gaivoronsky/Mistral-7B-Saiga)), квантует их (4 или 8 bit) и создаёт `config.json`.
2. **RAG-индекс**: `create_rag_index()` принимает CSV/XLSX/JSON/TXT, нормализует тексты так же, как и онлайн-сервис, и формирует артефакты:
   ```
   rag/
   ├── meta.json
   ├── index.faiss
   └── chunks.parquet
   ```
   Папка может находиться в `rag/`, `artifacts/rag/` или `data/rag/` — `FactualModel` сам ищет артефакты.

### Запуск из командной строки

```bash
python solution.py \
  --input questions.csv \
  --output answers.csv \
  --format csv \
  --batch-size 8 \
  --config config.json
```

Аргументы:
- `--input` — путь к файлу (JSON, CSV, XLSX, TXT)
- `--output` — куда сохранить результат
- `--format` — `json`, `json_pairs` или `csv`
- `--batch-size` — размер батча
- `--config` — путь к конфигурации (создаётся автоматически `download_model`)

## 🔧 Конфигурация

### Параметры RAG

Все значения берутся из `config.json`, который генерируется автоматически и может выглядеть так:

```json
{
  "model": {
    "name": "Gaivoronsky/Mistral-7B-Saiga",
    "dir": "./model/",
    "quantize": true,
    "bits": 8
  },
  "rag": {
    "dir": "./rag/",
    "top_k": 3,
    "threshold": 0.38,
    "max_ctx_chars": 1400,
    "embedding_model": "sentence-transformers/all-MiniLM-L12-v2"
  },
  "generation": {
    "max_new_tokens": 48,
    "seed": 127
  }
}
```

При необходимости параметры можно переопределить прямо в коде, передав `config_overrides` в конструктор `FactualModel`.

## 📊 Как это работает

### Процесс обработки запроса

1. **Быстрый путь (математика)**: Если запрос содержит математическое выражение, ответ вычисляется напрямую
2. **RAG путь**: 
   - Запрос нормализуется и лемматизируется
   - Выполняется векторный поиск в FAISS индексе
   - Если лучший score ≥ `DECISION_THRESHOLD`, используется найденный контекст
   - Генерируется ответ с учетом контекста
3. **Fallback путь**: Если RAG недоступен или score низкий, используется обычная генерация без контекста

### Обработка ответов

Система включает несколько уровней постобработки:
- Удаление эхо-ответов (повторение вопроса)
- Проверка на "сломанные" ответы
- Ограничение длины ответа
- Специальная обработка переводов

## 🎯 Особенности

- **Автоматическое определение GPU**: при наличии A100 (или любого CUDA) всё вычисляется на GPU, иначе включается fallback на CPU.
- **Гибкое квантование**: `download_model()` поддерживает 4-bit NF4 и 8-bit INT8 (через bitsandbytes).
- **RAG builder**: единая функция подготовки индекса с лемматизацией и нормализацией.
- **Батч-обработка**: `generate_batch()` работает на любых входных форматах.
- **Единые docstring'и**: все публичные функции имеют русскоязычные описания — удобно наводиться в IDE.
- **Табличный вывод**: ноутбук формирует таблицу ответов «как у @saiga_benchmark».

## 🔍 Примеры использования

## 🐛 Решение проблем

### RAG не инициализируется

Проверьте наличие всех файлов в директории `rag/`:
- `meta.json`
- `index.faiss`
- `chunks.parquet`

Система автоматически ищет файлы в нескольких альтернативных локациях.

### Проблемы с CUDA

Если возникают ошибки CUDA, система автоматически переключится на CPU режим. Для принудительного использования CPU:

```python
import os
os.environ['CUDA_VISIBLE_DEVICES'] = ''
```

### Недостаточно памяти

Уменьшите `batch_size` в `generate_batch()` или используйте CPU режим.
